# Asset Management System

A REST API for the DarkAtlas Attack Surface Monitoring platform — Track A
(Backend Engineering). Manages discovered assets (domains, subdomains, IPs,
services, certificates, technologies), deduplicates them on ingest, tracks
their lifecycle, and models the relationships between them.

## Stack

Python · FastAPI · SQLAlchemy · PostgreSQL · Docker Compose

## Running the project

### With Docker (recommended)

```bash
cp .env.example .env
docker compose up --build
```

The API will be available at `http://localhost:8000`.
Interactive API docs (Swagger UI): `http://localhost:8000/docs`.

### Locally without Docker

1. Start a PostgreSQL instance and create a database named `asset_management`.
2. Copy `.env.example` to `.env` and fill in `DATABASE_URL` / `API_KEY`.
3. Install dependencies and run:

```bash
pip install -r requirements.txt
uvicorn app.main:app --reload
```

## Environment variables

| Variable       | Description                                              |
|----------------|------------------------------------------------------------|
| `DATABASE_URL` | PostgreSQL connection string                                |
| `API_KEY`      | Shared secret required (via `X-API-Key` header) on all write operations |

## Authentication

Write operations (`POST`, `PUT`, `DELETE` on `/assets` and `/relationships`)
require an `X-API-Key` header matching the configured `API_KEY`. Read
operations (`GET`) are open.

```bash
curl -X POST http://localhost:8000/assets \
  -H "X-API-Key: change-me" \
  -H "Content-Type: application/json" \
  -d '{"type": "domain", "value": "example.com", "source": "scan"}'
```

## API overview

| Method | Path                       | Auth | Description |
|--------|-----------------------------|------|--------------|
| POST   | `/assets`                   | ✅   | Create an asset |
| POST   | `/assets/import`             | ✅   | Bulk import assets from a JSON file |
| GET    | `/assets`                   |      | List assets — filter by `type`, `status`, `tag`, `value`; sort via `sort_by`/`sort_order`; paginate via `skip`/`limit` |
| GET    | `/assets/certificates`       |      | List certificate assets filtered by lifecycle `cert_status` (`expired` / `expiring_soon` / `valid`) |
| GET    | `/assets/{id}`               |      | Get a single asset |
| GET    | `/assets/{id}/graph`         |      | Get an asset together with its related assets |
| PUT    | `/assets/{id}`               | ✅   | Update an asset |
| POST   | `/assets/{id}/mark-stale`    | ✅   | Lifecycle: mark an asset as stale |
| DELETE | `/assets/{id}`               | ✅   | Delete an asset |
| POST   | `/relationships`             | ✅   | Create a relationship between two assets |
| GET    | `/relationships/{asset_id}`  |      | List relationships for an asset |

Full interactive documentation is auto-generated by FastAPI at `/docs`.

## Error responses

Every error — validation failures, 404s, 401s, and unexpected server
errors — is shaped the same way:

```json
{
  "error": {
    "code": "NOT_FOUND",
    "message": "Asset not found"
  }
}
```

Validation errors additionally include a `details` array (Pydantic's
per-field error list):

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "One or more fields failed validation.",
    "details": [{"loc": ["body", "type"], "msg": "...", "type": "enum"}]
  }
}
```

## Design decisions & assumptions

- **Deduplication key**: assets are deduplicated by their `value` field
  (e.g. the domain name, IP, or cert CN). Re-importing an asset with the
  same value updates `last_seen`, reactivates it if it was `stale`, and
  merges `tags` (union) and `metadata` (shallow merge, new values win).
- **Lifecycle**: `first_seen` is set once on creation and never changes.
  `last_seen` updates on every re-sighting (create or import). Assets are
  marked `stale` explicitly via `POST /assets/{id}/mark-stale` (or via
  `PUT /assets/{id}` with `{"status": "stale"}`); a stale asset that is
  re-sighted automatically returns to `active`.
- **Relationships**: directional (`source_asset_id` → `target_asset_id`)
  with a free-text `relationship_type` (e.g. `parent`, `covers`,
  `resolves_to`). `GET /assets/{id}/graph` returns the asset plus every
  asset connected to it in either direction.
- **Bulk import**: processes records independently — a malformed record
  increments `failed` and the rest of the batch still completes.
  Relationships declared via `parent`/`covers` fields in the sample
  dataset are created in a second pass, after all assets in the batch
  have been resolved to database IDs.
- **Authentication**: kept intentionally lightweight (static API key via
  `X-API-Key` header) per the task's "lightweight authentication" scope.
  Not multi-tenant; all assets currently live in a single shared space.
- **Sorting**: defaults to `last_seen desc` so the most recently active
  assets surface first.
- **Certificate lifecycle**: a certificate's expiry date lives in
  `metadata_json["expires"]` (ISO date string, matching the sample
  dataset format). `cert_status` (`expired` / `expiring_soon` / `valid`)
  is computed on read — not stored — by comparing that date to "now",
  with a 30-day "expiring soon" window. It's exposed as a computed field
  on every asset response and as a filter via
  `GET /assets/certificates?cert_status=expired`.

## Running tests

Tests run against a real PostgreSQL instance (the model uses
PostgreSQL-specific column types — `UUID`, `JSONB`, `ARRAY` — so SQLite is
not used for testing). Point `DATABASE_URL` at a disposable test database
before running:

```bash
export DATABASE_URL=postgresql://postgres:1234@localhost:5432/asset_management_test
export API_KEY=test-api-key
pytest -v
```

Tests cover: CRUD, authentication on write endpoints, deduplication on
create and on bulk import, lifecycle (stale → active on re-sighting),
bulk import (including malformed records and idempotency), filtering,
sorting, pagination, and the relationship graph endpoint.
